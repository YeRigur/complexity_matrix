Lvl
Definition
Typical Examples
1
One change. A single fix, update, or addition in one place.
  • Change one field on one content type
  • Fix one template
  • Update one config file
  • Adjust one component's styles
2
A few changes. The same or similar work repeated in several spots.
  • Update the same field across several content types
  • Adjust styles on a few pages
  • Apply a config change to several multisite instances
  • Change a template and its preprocess function
3
Many changes. Lots of individual changes across the codebase.
  • Apply a change across many sites
  • Update many content types, each with its own specifics
  • Rework templates across an entire theme
  • Touch many modules or components in one task
Other
Doesn't match 1–3. Too small, too large, or hard to categorize.
  • One-character typo fix (too small to rate)
  • Platform-wide rebranding across all sites (too large to rate)
  • Ongoing support without clear deliverables
Lvl
Definition
Typical Examples (FE | BE)
1
A small fix with low unknowns and a clear solution.
Frontend
  • Update markup in an existing template (add class, swap element)
  • Swap an icon, image, or static asset
  • Fix a single responsive/adaptivity issue for one breakpoint or device
  • Adjust existing component's text, colour, or visibility via config
Backend
  • Fix a backend bug — correct a field value, condition, or config
  • Install and configure a contrib module via admin UI
  • Add or rename a field on a content type
  • Adjust display mode, View filter, or pager settings
  • Update validation rule or error message on an existing form
2
A feature or change with some unknowns and a generally known approach, possibly involving non-trivial business logic.
Frontend
  • Implement a UI component from Figma (responsive, cross-browser)
  • Add interactivity to a component (accordion, tabs, modal)
  • Consume a REST endpoint and render data with states
  • Integrate a frontend library within one component scope
Backend
  • Implement server-side logic for one feature (validation, data processing)
  • Build a Drupal plugin or module for a single responsibility
  • Create a new route + controller with a form
  • Extend a contrib module (field widget, condition plugin)
  • Fix a bug involving complex business logic
  • Implement access control or permissions for one resource
  • Set up an event subscriber or hook for a specific trigger
3
A complex feature with high unknowns that requires investigation, design decisions, or involves complex business logic.
Frontend
  • Build a reusable component library or design system
  • Stateful UI with multiple data sources (filters + AJAX + pagination)
  • Full CMS theming integration (template bindings, component architecture)
  • Front-end performance optimisation (loading, rendering, assets)
  • Integrate a 3rd-party frontend library with custom adapter layer
  • Work with an unfamiliar technology or framework not previously used in the project
Backend
  • Design and implement a REST API with contracts and error handling
  • Integrate with an external service or internal platform
  • Multi-service module (importer + processor + logger)
  • Content migration (Drupal Migrate API with custom source/process plugins)
  • Queue worker, cron job, or Batch API processing pipeline
  • Implement robust business logic with multiple conditions and rules
  • Investigate and untangle complex existing code or process
  • Refactor module architecture (split monolith, extract services)
Other
Doesn't match 1–3. Too simple, too complex, or not development work.
  • Pure content entry or copywriting (simpler than 1)
  • Drupal core upgrade or major platform rearchitecture (harder than 3)
  • Research or PoC without a clear deliverable
